Базовый процесс получения тикета с использованием фильтра и AD.
  1) Если в системе установлен конфигурационный параметр браузер (JS-код) делает запрос к фильтру. При этом автоматически предоставляются заголовки аутентификации   из браузера.
  2) Фильтр делает запрос к контроллеру домена с информацией из заголовков.
  3) AD возвращает имя пользователя, для которого верны предоставленные браузером заголовки.
  4) Фильтр делает доверенный запрос тикета для подтвержденного имени пользователя. Доверенным этот запрос делает полученный NTLM-фильтром при старте тикет локального пользователя веды (VedaNTLMFilter) с полномочиями получения тикетов для других пользователей.
  5) Veda создает и возвращает тикет.
  6) NTLM-фильтр возвращает тикет в браузер.

Альтернативный процесс - получение тикета напрямую из веды.
  7) Запрос аутентификации с локальным логином и паролем пользователя веды.
  8) Веда возвращает тикет.

Вариации основного процесса:
  Если автоматическая аутентификация с заголовками не прошла, NTLM предлагает браузеру заполнить логин и пароль вручную. Появляется стандартное браузерное окошко для ввода имени пользователя и пароля.
  а) Пользователь может заполнить доменные логин и пароль в стандартное браузерное окошко. Тогда повторяются все шаги базового процесса, только теперь NTLM -фильтр предоставляет домену пару логин:пароль.
  б) Пользователь отказывается вводить пароль в стандартное окошко. Тогда клиентский JS рисует нашу собственную форму аутентификации.
  Пользователь вводит логин:пароль. Клиентский JS делает специальный запрос к NTLM-фильтру. Который далее действует как в варианте а)
  Если в результате такого обращения аутентификацию пройти не удалось, клиентский JS-код делает запрос с предоставленными логином и паролем уже к серверу веды (стрелочки 7-8).

1) Для взаимодействия стороннего ПО с ведой, наиболее простой способ (в первую очередь для сторонней системы) - получить учетную запись в веде с бессрочным паролем, и статически прописать параметры для получения тикета напрямую от системы.
2) Равная альтернатива - получить доменную учетную запись в веде с бессрочным паролем и делать специальный запрос для получения тикета к NTLM-фильтру.
3) Еще одна альтернатива - доменная учетка с временным паролем. Наименее удобный способ, поскольку надо править конфигурационные параметры аутентификации (пароль).
